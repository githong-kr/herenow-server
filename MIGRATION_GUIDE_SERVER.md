# HereNow Server 마이그레이션 및 지원 가이드

이 문서는 공간적 기억 기법(Spatial Mnemonic)을 활용한 재고/물건 관리 프론트엔드를 지원하기 위해 백엔드(`herenow-server`)에서 구조를 개편하는 가이드라인입니다.

## 🏗 전체 프로젝트 개편 기조 (공통)
1. **아키텍처:** 렌더링 주체가 웹뷰(Vite/React) 환경으로 완전히 이동했습니다. localhost 환경에서의 원활한 CORS 처리가 필수적입니다.
2. **명확한 데이터 계층 (3-Tier Structure):** 1차원적인 리스트 배열을 넘어, `[전체 구역(Room)] ➡️ [세부 수납장(Furniture)] ➡️ [보관 물품(Item)]` 형태의 논리적인 3단 트리 구조로 스키마를 재구성합니다.
3. **유틸리티 중심주의:** 방 크기(width/height)는 프론트엔드의 시각적 배치를 돕기 위한 보조 데이터일 뿐, 시스템적인 수납량을 강제 제한하는 물리 엔진은 배제합니다. 가장 중요한 것은 "사용자가 입력한 재고와 유통기한 데이터"의 정합성입니다.

---

## 💾 Server 역량 확장의 핵심 (Spatial Data & Tree Relation)
물건의 위치 데이터가 이전보다 훨씬 명확해집니다. 물품 테이블은 상위 가구(Furniture)의 ID를 바라보며, 가구는 상위 방(Room)의 ID를 바라보는 완벽한 계층형 조회성을 지녀야 합니다. 특히 프리미엄 비즈니스 파트와 알림 시스템을 담당하게 됩니다.

---

## ✅ Server 작업 체크리스트 (Tasks)

### Step 1: 3계층(3-Tier) 관리를 위한 Database 스키마 고도화
- [ ] **Rooms (구역 테이블):** 
  - 도면 맵 상에서의 시각적 좌표(`grid_x`, `grid_y`)와 크기(`width`, `height`) 보관용 껍데기 역할
- [ ] **Furnitures (수납장/가구 테이블 - 신설):**
  - 속해 있는 구역(`room_id`) FK 참조
  - 방 내부의 상대적 뷰 배치를 위한 보조 좌표 필드(옵션) 추가
  - **BM (비즈니스 수익화) 연동**: 제휴용 브랜드 프리셋 가구/템플릿(예: LG 비스포크 냉장고) 등을 지원하기 위한 메타데이터 혹은 `template_id` 필드를 추가.
- [ ] **Items (개별 물품 테이블):**
  - 속해 있는 수납장(`furniture_id`) FK 참조로 마이그레이션
  - 물건 이름, 수량(Quantity), 상태, **유통기한(Expiration Date)** 등 핵심 관리 지표를 다루는 필드 강화
  - 템플릿 그룹핑 지원 (예: '국룰 냉장고 세팅'용 프리셋 식별자 추가)

### Step 2: 클라이언트 퍼포먼스를 및 소셜 알림 (Notification/Share) API
- [ ] 전체 3단 구조 트리를 한 번의 호출로 뿌려주거나, 구조 최적화를 거친 DTO (Lazy Loading 지원)
- [ ] 위치 이동을 위한 단일 PATCH API (거실 선반 -> 주방 냉장고 스위칭 지원)
- [ ] **알림/푸시 스탯 (Notification):** 유통기한이 임박(D-Day 1~3일)한 물품들만 조회, 클라이언트 진입 전에도 앱 푸시나 뱃지(Badge) 형태의 정보를 제공하는 스탯 API 구축
- [ ] **공유 (Multi-user Sync):** 가족 단위 등 같은 디오라마(냉장고 상태 등)를 실시간으로 공유하고 조회 및 편집할 수 있도록 Supabase/Firebase 또는 WebSocket 기반의 실시간 API 구조 검토

### Step 3: 개발 공통 인터페이스
- [ ] 로컬 WebView 개발망(`localhost:5173`) 연동을 위한 CORS 정책 완화 및 로컬 API 지원
- [ ] Web OpenAPI generator용 스키마 JSON 문서 명확화 및 유지 보수

## 🔍 자급자족 품질 검토 프로세스 (Self-QA Protocol)
새로운 3계층 트리 구조의 병목을 중점적으로 테스트합니다.
1. **API 호출 빈도 점검:** 물건 하나를 추가할 때마다 전체 트리를 Re-fetch 해야 하는지, 상태가 부분 업데이트 될 수 있는지 확인.
2. **사용자 유틸성 관점:** 유저가 진짜 찾고자 하는 것은 방의 위치보다는 "유통기한 지난 계란"입니다. 데이터 모델이 시각적 놀이에 매몰되지 않고 검색 기능을 원활히 수행하도록 인덱스를 설계합니다.
